# Translation Brief (en → de)

## 1) App Context
- **Product type:** Developer-focused SaaS/open-source tool landing page.
- **Domain:** Localization/i18n automation for web apps.
- **Core value prop:** AI-assisted translation workflow for **React/Next.js** with **CLI**, CI support, and no manual key-file maintenance.
- **Target audience:** Developers, tech leads, and engineering teams (comfortable with technical terminology).

---

## 2) Register & Formality
- Use **informal “du”** consistently.
- Address users directly in CTA copy with **active verb forms**.
- Do **not** mix with “Sie”.

**Examples**
- “Get Started” → **„Jetzt starten“** / **„Starte jetzt“**  
- “Ready to go global?” → **„Bereit, global zu gehen?“**

---

## 3) Tone
- **Professional + technical + concise**, with light marketing energy.
- Confident and practical, not playful.
- Focus on speed, automation, and developer efficiency.
- Avoid buzzword-heavy or overly literal translations.

---

## 4) Term Decisions

### Keep in English (do not translate)
- **Brand/product names:** `Tyndale`, `GitHub`
- **Framework/tool names:** `React`, `Next.js`, `JSX`, `CLI`, `CI`, `RTL`
- **Command names/code:** `tyndale validate` (exact spelling)
- **Common industry term:** `i18n`
- **Established tech term:** `Open Source` (capitalize consistently)

### Translate to German
- Marketing/action copy and general UI text:
  - “Get Started” → „Jetzt starten“
  - “View on GitHub” → „Auf GitHub ansehen“
  - “Write your app in one language.” → „Schreibe deine App in einer Sprache.“
  - “See the result” → „Sieh dir das Ergebnis an“
- Prefer **KI-** for “AI-” in German:
  - “AI-powered” → „KI-gestützt“
- Avoid literal mistranslations:
  - “Ship translated.” should be **„Lokalisiert ausliefern“** (not literal shipping wording)

### Preferred German renderings for tricky items
- “Zero-key workflow” → **„Workflow ohne Key-Dateien“** (or „ohne Schlüsseldateien“, if less technical)
- “Next.js first-class” → **„Erstklassige Next.js-Integration“**
- “CI-friendly” → **„CI-freundlich“**
- “Switch locales” → **„Locales wechseln“** (developer-facing), optional broader UI: „Sprachvarianten wechseln“

---

## 5) Patterns

### CTAs
- Use **short, action-first phrasing**.
- Primary CTAs: imperative or compact action phrase.
- Keep to 1–4 words where possible.
- No period at end of button labels.

**Examples**
- „Jetzt starten“
- „Auf GitHub ansehen“
- „Übersetzen“
- „Installieren“

### Error messages
- Structure: **Problem + next step**.
- Tone: neutral, helpful, non-blaming.
- Keep technical identifiers/commands unchanged.

**Pattern**
- „[Aktion] konnte nicht abgeschlossen werden. [Nächster Schritt].“

### Empty states
- Structure: **Current state + suggested action**.
- Keep concise and constructive.

**Pattern**
- „Noch keine Übersetzungen vorhanden. Starte mit [Aktion].“

### Confirmations/success
- Short and outcome-focused.
- Optional follow-up action if useful.

**Pattern**
- „Übersetzung abgeschlossen.“ / „Validierung erfolgreich.“

### Questions/headlines
- Use direct **du** voice.
- Keep rhetorical questions short and confident.
- Avoid over-punctuation/exclamation overload.

**Example**
- „Bereit, global zu gehen?“
